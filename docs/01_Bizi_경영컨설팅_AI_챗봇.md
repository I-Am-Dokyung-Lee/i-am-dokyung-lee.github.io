## 담당 역할 상세

### 1. 데이터 전처리 파이프라인 구축

비정형 법령/공고 데이터를 RAG 파이프라인에 적재할 수 있도록 전처리 파이프라인을 설계하고 구축했습니다.

- **원본 데이터**: 법제처 법령 XML, 정부 공고 HTML/PDF 등 비정형 문서
- **처리 파이프라인**: 문서 파싱 → 노이즈 제거 → 도메인별 분류 → 메타데이터 태깅 → 청크 분할 → 벡터 임베딩
- **처리 규모**: 4개 도메인, 21만 건 이상의 문서 데이터
- **청킹 전략**: 법령 특성에 맞춘 조항 단위 분할, 오버랩 설정으로 문맥 손실 최소화

### 2. Redis 기반 멀티턴 대화 처리

단일 질문-응답이 아닌, 경영 컨설팅에 필수적인 **연속 대화**를 가능하게 하는 세션 관리 시스템을 구현했습니다.

- **Redis 세션 메모리**: 사용자별 대화 히스토리를 Redis에 저장, TTL 정책으로 자동 만료
- **대화 맥락 요약**: 토큰 제한 내에서 이전 대화의 핵심 맥락을 유지하는 요약 파이프라인
- **토큰 관리**: 긴 대화 시 컨텍스트 윈도우 초과 방지를 위한 슬라이딩 윈도우 + 요약 전략

### 3. 보안 감사

프로덕션 배포 전 보안 점검을 수행하여 **CRITICAL/HIGH 16건의 취약점**을 수정했습니다.

- **관리자 권한 탈취 차단**: 관리자 API 엔드포인트에 대한 인증/인가 미비를 발견하고 권한 검증 미들웨어 추가
- **소유권 검증**: 다른 사용자의 리소스에 접근 가능한 IDOR 취약점을 수정, 모든 데이터 접근에 소유권 검증 로직 추가
- **SQL Injection 방어**: Raw SQL 사용 부분을 파라미터화 쿼리로 전환

### 4. RAGAS 평가 시스템 개선

RAG 성능을 정량적으로 측정하고 개선하기 위한 평가 시스템을 구축했습니다.

- **한국어 프롬프트 최적화**: RAGAS 기본 영어 프롬프트를 한국어에 맞게 커스터마이징하여 평가 정확도 향상
- **배치 평가 CLI**: 80개 테스트 질문을 일괄 평가하는 CLI 도구 개발, 반복 실험 효율화
- **5단계 반복 개선**: 청킹 전략, 프롬프트, 검색 파라미터를 단계별로 조정하며 정량적 성능 추적

## 문제 해결 사례

### 사례 1: 멀티턴 대화 맥락 유지

**상황**
초기 시스템은 단일 질문-응답만 가능했습니다. 경영 컨설팅은 "법인 설립 절차가 궁금해요" → "그러면 필요한 서류는?" → "비용은 어느 정도인가요?"처럼 연속적인 대화가 필수적인데, 매 질문마다 맥락이 초기화되어 실제 상담 용도로 사용이 불가능했습니다.

**시도 및 해결**
- Redis를 세션 스토어로 선택 (인메모리 DB의 빠른 읽기/쓰기, TTL 자동 만료)
- 대화 히스토리를 사용자 세션별로 저장하되, 토큰 제한을 고려한 요약 파이프라인 추가
- 이전 대화의 핵심 엔티티(회사명, 업종, 이전 질문 주제)를 추출하여 후속 질문의 RAG 검색 쿼리에 반영

**결과**
사용자가 이전 질문을 참조하며 심화 상담이 가능해졌고, "그러면", "추가로" 등의 연속 질문에서도 맥락을 정확히 유지. 실제 경영 컨설턴트와 대화하는 것에 가까운 UX를 구현했습니다.

### 사례 2: RAG 파이프라인 성능 고도화

**상황**
초기 RAG 시스템의 RAGAS 평가 결과가 Faithfulness 0.36, Context F1 0.35로 답변 신뢰성이 매우 낮았습니다. 법령/세무 도메인은 정확한 출처 기반 답변이 필수적인데, 관련 없는 문서를 검색하거나 LLM이 근거 없는 답변을 생성하는 문제가 빈번했습니다.

**시도 및 해결**
- 청킹 전략 변경: 일률적 토큰 분할 → 법령 조항 단위 분할로 문맥 보존
- 한국어 RAGAS 프롬프트 최적화: 영어 기본 프롬프트가 한국어 평가에서 부정확한 결과를 내는 문제 해결
- Cross-Encoder 재순위화 적용: 1차 검색 결과를 Cross-Encoder로 재정렬하여 관련성 높은 문서 상위 배치
- 배치 평가 CLI 구축: 80개 질문 일괄 평가로 매 변경마다 정량적 성능 비교 가능

**결과**

| 평가지표 | 초기(A) | 최종(E) | 개선율 |
|:---:|:---:|:---:|:---:|
| Faithfulness | 0.36 | 0.61 | +72.5% |
| Answer Relevancy | 0.61 | 0.68 | +11.9% |
| Context Precision | 0.54 | 0.74 | +37.0% |
| Context F1 | 0.35 | 0.63 | +79.7% |
| 거부 정확도 | 0% | 100% | 완전 달성 |

| 기술 | 선택 이유 | 대안 대비 장점 |
|:---|:---|:---|
| **LangGraph** | 4개 도메인 분류 → 검색 → 생성 → 평가의 복잡한 분기 로직을 상태 머신으로 관리 | LangChain 선형 체인 대비 조건부 분기, 루프, 병렬 처리 지원 |

## 시스템 아키텍처

```
[Client - React 18]
        ↓ HTTPS
[Nginx - Reverse Proxy / SSL]
        ↓
[Backend - FastAPI]  ←→  [MySQL 8.0 (RDS)]
        ↓                      ↓
[RAG Service]           [Redis (ElastiCache)]
   ↓        ↓
[ChromaDB] [RunPod GPU Serverless]
                ↓
        [AWS S3 - 문서 저장]
```

- AWS EC2 위에서 Docker Compose로 6개 컨테이너 운영
- MSA 구조: Frontend / Backend / RAG 서비스 분리
- HTTPS 강제, HSTS, CSRF 보호, Rate Limiting 등 보안 적용

## RAG 파이프라인 (LangGraph 5단계)

| 단계 | 핵심 기술 | 설명 |
|:---:|:---|:---|
| CLASSIFY | LLM + Keyword Fallback | 4개 도메인 자동 분류 |
| DECOMPOSE | Multi-Query | 복합 질문을 서브쿼리로 분해 |
| RETRIEVE | BM25 + Vector + RRF | 하이브리드 검색 + Cross-Encoder 재순위화 |
| GENERATE | GPT-4o-mini | 기업 프로필 맥락 반영, 출처 명시, SSE 스트리밍 |
| EVALUATE | LLM Self-Eval | 품질 미달 시 멀티쿼리 재시도 |

## 회고

이 프로젝트에서 가장 크게 배운 점은 **"AI 시스템은 모델 성능만으로 완성되지 않는다"**는 것입니다.

데이터 전처리 단계에서 법령 조항 단위로 청킹 전략을 바꾼 것이 RAGAS 성능 향상의 가장 큰 요인이었고, 멀티턴 구현에서는 단순히 대화를 저장하는 것이 아니라 "어떤 맥락을 유지하고 어떤 것을 버릴지"의 판단이 핵심이었습니다.

보안 감사를 통해서는 "동작하는 코드"와 "프로덕션에 올릴 수 있는 코드" 사이의 간극을 체감했습니다. 특히 IDOR 취약점은 기능 테스트에서는 발견되지 않지만, 실제 서비스에서는 치명적인 문제가 될 수 있다는 것을 배웠습니다.
