## 프로젝트 동기

반려동물 건강 관련 정보는 인터넷에 넘치지만, 비전문적인 정보나 광고성 콘텐츠가 섞여 있어 **신뢰할 수 있는 정보를 빠르게 얻기 어려운 문제**가 있었습니다.
LLM의 내부 지식에 의존하면 의료 도메인에서 부정확한 답변(Hallucination)이 생길 수 있다는 점에 착안하여, **RAG 기반으로 검증된 데이터만 활용**하되, 필요 시 웹 검색을 보조적으로 사용하는 복합 추론 시스템을 설계했습니다.

---

## 프로젝트 소개

LangGraph 기반 복합 추론 Agent를 활용한 **반려견 건강 증상 분석 챗봇**입니다.
사용자가 반려견의 증상을 입력하면 RAG 기반으로 관련 질병 정보를 검색하고, 필요 시 웹 검색을 병행하여 종합적인 건강 상담을 제공합니다.

---

## LangGraph 6노드 워크플로우 설계

### 설계 의도

단순 RAG(질문 → 검색 → 답변)가 아닌, **의도 분류 → RAG 검색 → 결과 품질 판단 → 보조 검색 → 답변 생성**의 단계별 추론을 구현했습니다. 각 노드가 독립적인 판단을 수행하여, 질문 유형과 검색 결과 품질에 따라 동적으로 파이프라인이 분기됩니다.

### 6노드 구성 및 분기 조건

```
사용자 입력
  ↓
[1] Analyze_Intent ──→ HEALTH_RISK / GENERAL 분류
  ↓
[2] RAG_Search ──→ ChromaDB 벡터 검색
  ↓
[3] Determine_Action ──→ RAG 결과 충분? ──Yes──→ [5]
  │                                       └─No──→ [4]
  ↓
[4] Web_Search ──→ Google Custom Search API
  ↓
[5] General_LLM_Response ──→ 보조 응답 생성
  ↓
[6] Final_Generation ──→ RAG Guardrail 적용 최종 답변
```

| 노드 | 역할 | 분기 조건 |
|:---|:---|:---|
| Analyze_Intent | 질문 의도 분류 | HEALTH_RISK → RAG 우선 / GENERAL → LLM 직접 응답 |
| RAG_Search | 벡터 DB 검색 | 항상 실행 (HEALTH_RISK 시) |
| Determine_Action | RAG 결과 충분성 판단 | 충분 → Final 직행 / 부족 → Web 검색 추가 |
| Web_Search | 웹 보조 검색 | RAG 결과 부족 시에만 실행 |
| General_LLM_Response | 일반 응답 생성 | GENERAL 질문 시 |
| Final_Generation | Guardrail 적용 최종 답변 | 항상 실행 |

**Determine_Action 노드의 의의:** RAG 결과가 충분한 경우 불필요한 웹 API 호출을 건너뛰어 응답 속도를 최적화하면서도, 정보가 부족한 경우에는 자동으로 웹 검색으로 보완합니다.

## RAG Guardrail (차별화 포인트)

의료 도메인에서 LLM의 Hallucination은 잘못된 조치로 이어질 수 있어, **엄격한 Guardrail**을 프롬프트 엔지니어링으로 구현했습니다.

### Guardrail 규칙
1. **제공된 컨텍스트만 사용**: LLM 내부 지식 사용을 명시적으로 금지하는 프롬프트 구성
2. **구조화 답변 강제**: 모든 답변에 아래 포맷을 강제 적용
   - 수의사 방문 권유 (해당 시)
   - 증상/위험도 분석
   - 예상 원인 (출처 명시)
   - 권장 조치
3. **불확실성 명시**: 충분한 정보가 없을 경우 "정보 부족" 또는 "수의사 상담 필요"를 우선 표시

### 프롬프트 설계 핵심

```python
# Guardrail 프롬프트 핵심 구조 (실제 코드 스니펫)
system_prompt = """
당신은 반려견 건강 상담 전문가입니다.
반드시 아래 규칙을 따르세요:
1. 제공된 [참고 자료]의 내용만을 기반으로 답변하세요.
2. [참고 자료]에 없는 내용은 절대 답변하지 마세요.
3. 정보가 부족한 경우 "수의사 상담을 권장합니다"라고 답변하세요.
4. 답변 형식: 증상 분석 → 예상 원인 → 권장 조치 → 출처
"""
```

## 데이터

- **원천데이터**: 말뭉치데이터 (내과, 안과 등) - disease 필드를 청크 분할하여 RAG 검색 대상으로 사용
- **라벨링데이터**: 질의응답데이터 - lifeCycle, department, disease 메타데이터로 활용
- **벡터화**: ChromaDB에 저장, 청크 크기 512 / 오버랩 50
